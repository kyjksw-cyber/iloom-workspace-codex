# 일룸 매출 통합 대시보드 PRD

> 생성일: 2026-03-12
> 생성 방법: `/dashboard-prd 일룸 매출 통합` (대화형 4-Phase)

---

## 1. 프로젝트 개요

### 핵심 의사결정

이 대시보드를 보고 내리는 결정:

1. **채널 전략 조정**: 오프라인/네이버/오늘의집/공식몰 중 어디에 더 투자할지 (매출 + 수익성 기반)
2. **매장 관리**: 목표 미달 매장에 프로모션을 걸지, 인력을 재배치할지
3. **상품 운영**: 다음 달 발주량 조정, 온라인 전용 상품 확대 여부
4. **수익성 판단**: 같은 제품을 어디서 팔면 마진이 가장 좋은가 → 채널별 가격/수수료 정책 조정

### 배경 (Pain Points)

- ERP에서 엑셀을 내려받아 직접 정리하고, 구글시트 연결에서 막힘 (한병화)
- 태블로로 1차 가공 후 분석 도구에 넣는 이중 작업 (최지연)
- 네이버/오늘의집/공식몰 데이터 포맷이 전부 달라 통합이 복잡 (온라인영업팀)
- 같은 상품인데 채널별 코드가 3종류 (NV-/OH-/내부코드)
- 매번 자료 가공에 2~4시간 소요
- 매출만 보고 수익성(채산)은 별도 분석 필요 (안권희)

### 대시보드 유형 판정

- **열람자**: 팀장들이 매일 + 경영진이 주 1회 회의 → **전술 대시보드**
- **특성**: 일간~주간 갱신, 액션 연결, 드릴다운 가능
- **KPI 수**: 5~7개

### 목적

"데이터 넣으면 바로 보인다" -- 4개 채널 원본 데이터 가공부터 대시보드 표시까지 전 과정을 자동화하여, 채널/매장/상품 의사결정을 데이터 기반으로 내린다.

### 핵심 가치

1. **통합**: 4개 채널 매출을 하나의 화면에서 비교
2. **자동화**: 원본 파일 → 정제 → 시트 → 대시보드 파이프라인을 스킬 한 줄로 실행
3. **수익성**: 단순 매출이 아닌 채널별 마진/수수료를 반영한 채산 분석
4. **반복성**: 매월 새 데이터가 들어와도 같은 프로세스로 즉시 반영

## 2. 데이터 소스

모든 원본 데이터는 `50-resources/raw-data/`에 위치한다.

### 소스 1: ERP 오프라인 수주 (`erp-매출-YYYYMM.xlsx`)

- **위치**: `raw-data/erp-매출-202601.xlsx` ~ `erp-매출-202603.xlsx` (월별 분리)
- **형태**: xlsx (시트명: 수주현황)
- **규모**: 50~60행/월 x 20컬럼
- **특이사항**:
  - Row 1: 제목, Row 2: 출력일시, Row 3: 빈 행, Row 4-5: 병합 헤더 2줄
  - 마지막 행: 합계 (순번이 숫자가 아닌 행)
  - 날짜 형식: `2026-01-05`
- **필요 컬럼**: 수주일자, 대리점코드, 품목코드, 품목명, 시리즈, 카테고리, 수량, 단가(공급가), 소비자가, 할인율, 수주금액, 배송상태
- **정제 규칙**:
  - Row 1~3 제거, Row 5를 컬럼명으로 사용
  - 마지막 합계 행 제거
  - 배송상태 "취소" 행 제외

### 소스 2: 네이버 스마트스토어 (`네이버-주문내역-YYYYMM.xlsx`)

- **위치**: `raw-data/네이버-주문내역-202601.xlsx`
- **형태**: xlsx (시트명: 주문조회)
- **규모**: ~80행/월 x 30컬럼
- **특이사항**:
  - 컬럼 30개 중 필요한 건 8개
  - 상품코드: NV- 접두어 (NV-LP1200)
  - 날짜: `2026-01-24 21:10:00` (datetime)
  - 금액: 숫자, 수수료 ~5.5%
- **필요 컬럼**: 주문일시, 상품번호, 상품명, 수량, 상품별 합계금액, 정산예정금액, 수수료, 주문상태
- **정제 규칙**:
  - 30개 → 8개 컬럼 추출
  - 주문일시에서 날짜만 추출 (시간 제거)
  - 주문상태 "취소" 행 제외
  - NV코드 → 상품기준시트로 일룸상품코드 매핑

### 소스 3: 오늘의집 (`오늘의집-주문현황-YYYYMM.xlsx`)

- **위치**: `raw-data/오늘의집-주문현황-202601.xlsx`
- **형태**: xlsx (시트명: 주문현황)
- **규모**: ~45행/월 x 20컬럼
- **특이사항**:
  - 상품코드: OH- 접두어 (OH-LP1200)
  - 날짜: `2026.01.20` (점 구분)
  - **금액이 텍스트** ("1,890,000" -- 쉼표 포함 문자열)
  - 수수료: 10~15% (카테고리별)
- **필요 컬럼**: 주문일, 셀러상품코드, 노출상품명, 주문수량, 결제금액, 정산금액, 수수료율, 배송상태
- **정제 규칙**:
  - 금액 텍스트 → 숫자 (쉼표 제거)
  - 수수료율 "15%" → 0.15
  - 날짜 점(.) → 하이픈(-) 통일
  - OH코드 → 상품기준시트로 일룸상품코드 매핑

### 소스 4: 공식몰 (`공식몰-주문-YYYYMM.csv`)

- **위치**: `raw-data/공식몰-주문-202601.csv`
- **형태**: csv
- **규모**: ~35행/월 x 15컬럼 (영문)
- **특이사항**:
  - 컬럼명이 영문 (order_date, product_code, total_price 등)
  - 날짜: `01/05/2026` (미국식 MM/DD/YYYY)
  - 상품코드: 일룸 내부코드 그대로 (LP-1200)
  - 수수료 없음 (자사몰)
  - 주소에 쉼표 포함 (CSV 파싱 주의)
- **필요 컬럼**: order_date, product_code, product_name, qty, unit_price, total_price, discount, status
- **정제 규칙**:
  - 날짜 MM/DD/YYYY → YYYY-MM-DD
  - status "cancelled" 행 제외

### 기준 데이터: 상품기준시트 (`상품기준시트.xlsx`)

- **규모**: 13개 SKU (공통 10 + 온라인전용 3)
- **핵심 역할**: 채널별 상품코드 → 일룸상품코드 통합 매핑
  - NV-LP1200 → LP-1200
  - OH-LP1200 → LP-1200
  - 공식몰은 이미 일룸코드 사용
- **온라인 전용 상품** (ERP에 없음 -- 통합 시 누락 위험):

| 일룸코드 | 상품명 | 카테고리 | 소비자가 |
|----------|--------|----------|----------|
| MN-QM | 모나코 퀸 매트리스 | 침실 | 690,000 |
| PT-D120 | 포테 데스크 1200 | 서재 | 359,000 |
| SN-SC | 시에나 사이드체어 | 거실 | 189,000 |

### 기준 데이터: 매장마스터 (`매장마스터.xlsx`)

- **규모**: 15개 매장
- **컬럼**: 대리점코드, 매장명, 지역, 매장유형, 개설일, 면적(평), 담당자
- **지역 분포**: 서울 3, 경기 4, 부산 2, 대구 1, 인천 1, 광주 1, 대전 1, 제주 1, 강원 1

### 기준 데이터: 목표실적 (`목표실적.csv`)

- **형태**: 피벗 (월이 가로 배치)
- **컬럼**: 매장코드, 2026-01, 2026-02, 2026-03
- **주의**: "매장코드" = ERP의 "대리점코드" (동일 값, 컬럼명만 다름)
- **정제**: unpivot (매장코드, 월, 목표금액) + 컬럼명 "매장코드" → "대리점코드" 통일

## 3. 데이터 파이프라인

### 전체 흐름

```
원본 파일 (raw-data/)
    |
[Step 1] 구조 파악 & 정제
    - ERP: 병합헤더/합계행 제거, 취소 제외
    - 네이버: 30 → 8컬럼 추출, 날짜 정리
    - 오늘의집: 텍스트 → 숫자, 날짜 통일
    - 공식몰: 날짜 변환, 취소 제외
    |
[Step 2] 코드 매핑
    - 상품기준시트 기준 채널별 코드 → 일룸상품코드
    - NV-LP1200 / OH-LP1200 / LP-1200 → LP-1200
    |
[Step 3] 채널 통합
    - 4개 소스를 단일 테이블로 병합
    - 채널 구분 컬럼 추가 (ERP/네이버/오늘의집/공식몰)
    |
[Step 4] 매장/목표 조인
    - 대리점코드 → 매장마스터 (매장명, 지역)
    - 목표실적 unpivot 후 조인 (오프라인만)
    |
[Step 5] 채산(수익성) 계산
    - 채널별 수수료 반영 → 정산금액
    - 공급가 기준 마진 계산
    |
[Step 6] 구글시트 입력
    - 가공 완료 데이터를 구글시트에 기록
    - 시트 구조: 통합매출, 매장별요약, 채널별요약, 목표달성
    |
[Step 7] 대시보드 반영
    - 시트 데이터를 대시보드 API로 fetch
    - 브라우저 새로고침으로 반영
```

### 통합 테이블 스키마

| 컬럼명 | 타입 | 소스 | 설명 |
|--------|------|------|------|
| date | date | 각 채널 날짜 통일 | YYYY-MM-DD |
| month | string | date 파생 | YYYY-MM |
| channel | string | 소스 구분 | ERP/네이버/오늘의집/공식몰 |
| product_code | string | 상품기준시트 매핑 | 일룸상품코드 (LP-1200) |
| product_name | string | 상품기준시트 | 상품명 |
| series | string | 상품기준시트 | 시리즈 |
| category | string | 상품기준시트 | 카테고리 |
| qty | number | 각 채널 | 수량 |
| sale_amount | number | 매출금액 | ERP: 수주금액 / 네이버: 합계금액 / 오늘의집: 결제금액 / 공식몰: total_price |
| supply_price | number | 공급가 | ERP: 공급가 / 온라인: 소비자가 기준 |
| commission | number | 수수료 | ERP: 0 / 네이버: ~5.5% / 오늘의집: 10-15% / 공식몰: 0 |
| settled_amount | number | 정산금액 | 매출 - 수수료 |
| margin | number | 계산 | settled_amount - supply_price |
| margin_rate | number | 계산 | margin / sale_amount * 100 |
| store_code | string | ERP만 | 대리점코드 (온라인은 null) |
| store_name | string | 매장마스터 조인 | 매장명 (온라인은 채널명) |
| region | string | 매장마스터 조인 | 지역 (온라인은 "온라인") |

### 채산(수익성) 계산 근거

| 채널 | 매출금액 | 수수료 | 정산금액 | 마진 |
|------|----------|--------|----------|------|
| ERP(오프라인) | 수주금액 | 0 | = 수주금액 | 수주금액 - 공급가 |
| 네이버 | 합계금액 | ~5.5% | 정산예정금액 | 정산예정금액 - 추정원가 |
| 오늘의집 | 결제금액 | 10-15% | 정산금액 | 정산금액 - 추정원가 |
| 공식몰 | total_price | 0 | = total_price | total_price - unit_price |

**필터**: 매장유형 (프리미엄샵/브랜드샵), 지역

### View 3: 채산 분석 (수익성)

**결정**: "같은 제품을 어디서 팔면 마진이 가장 좋은가? 채널별 가격/수수료 정책을 조정해야 하나?"

**KPI 카드**:

| KPI | 현재값 예시 | 비교 기준 | 방향 | 출처 | 계산 근거 |
|-----|-------------|-----------|------|------|-----------|
| 전체 마진율 | ~32.4% | 전월 대비 | +0.8%p ↑ | ERP 수주 + 3사 정산 + 상품기준시트 | SUM(margin) / SUM(sale_amount) * 100 |
| 최고 마진 채널 | 공식몰 (~41.2%) | -- | -- | 공식몰 주문 + 상품기준시트 | 채널별 마진율 MAX |
| 최저 마진 채널 | 오늘의집 (~18.7%) | -- | -- | 오늘의집 정산 + 상품기준시트 | 채널별 마진율 MIN |

**차트**:

| # | 차트 유형 | 데이터 | 선택 이유 | 출처 |
|---|-----------|--------|-----------|------|
| 1 | 수직 막대 (그룹) | 채널별 마진율 비교 | 카테고리 비교(4개 채널) → 막대 최적 | 채널별 정산 + 상품기준시트 |
| 2 | 수평 막대 | 상품별 채널 간 수익성 비교 (같은 제품의 채널별 마진) | 순위 + 비교 → 수평 막대 | 채널별 정산 + 상품기준시트 |
| 3 | 꺾은선 | 채널별 마진율 월간 추이 | 시간 변화 → 꺾은선 | 채널별 정산 + 상품기준시트 |

**필터**: 기간 (월), 카테고리, 상품

### AI 인사이트 제안 패널

각 View 하단에 AI가 현재 데이터를 분석하여 "생각해볼 지점"을 최대 3개 제시한다. 결론이 아니라 데이터 기반 질문을 던져 의사결정을 돕는다.

**UI 위치**: 각 View 차트 아래, 카드형 패널 (접기/펼치기 가능)

**인사이트 유형 및 탐지 규칙**:

| 유형 | 탐지 조건 | View | 예시 |
|------|-----------|------|------|
| 이상 탐지 | 전월 대비 ±15% 이상 변화 | 1, 3 | "오늘의집 매출이 전월 대비 +23%인데, 마진율은 -2.1%p입니다. 수수료 구간이 변경되었을 수 있습니다." |
| 추세 경고 | 3개월 연속 같은 방향 변화 | 2 | "분당서현 매장이 3개월 연속 목표 미달입니다 (87% → 82% → 78%). 추세가 하락 중입니다." |
| 기회 발견 | 비중 vs 성과 불균형 | 1, 3 | "공식몰 마진율이 전 채널 최고(~41%)인데 매출 비중은 12%입니다." |

**설계 규칙**:
- 톤: 단정이 아닌 제안 ("~일 수 있습니다", "확인해볼 필요가 있습니다")
- 근거 필수: 모든 인사이트에 참조 숫자 명시
- 개수: View당 최대 3개
- 생성: 데이터 갱신 시(월 1회) LLM API로 인사이트 생성 → 정적 텍스트로 저장. 실시간 호출 아님

**구현**:
- 데이터 가공 스킬 실행 시 통합 데이터를 LLM에 전달하여 인사이트 텍스트 생성
- 생성된 인사이트를 JSON으로 저장 (`public/data/insights.json`)
- 대시보드는 정적 JSON을 읽어 표시 (추가 API 호출 없음)

## 5. 기술 스택

데이터 규모: 월 200~250건 x 12개월 = 최대 3,000행. 열람자: 팀장 + 경영진 약 5명.
→ **10,000행 미만 + 열람자 소수 + 갱신 수동(월간)** 조건에 해당.

| 기술 | 용도 | 선택 이유 |
|------|------|-----------|
| Next.js 14+ (App Router) | 대시보드 프레임워크 | ISR 캐싱으로 Sheets API quota 절약, SSR로 초기 로딩 최적화 |
| React 18 | UI 컴포넌트 | Next.js 기본 |
| shadcn/ui | 기본 컴포넌트 | Card, Badge, Tooltip, Table, Select. CSS 변수 기반이라 Section 4.5 디자인 토큰으로 브랜드 오버라이드. 스타일 가이드 없이도 세련된 기본값 |
| Recharts | 차트 라이브러리 | 선언적 API, 커스터마이징 용이, 번들 크기 적절 |
| Tailwind CSS | 스타일링 | 빠른 UI 구현, 반응형 유틸리티 |
| TypeScript | 타입 안전성 | 데이터 스키마 변경 시 컴파일 타임 검증 |
| Google Sheets API | 데이터 소스 | 3,000행 + 5명 열람 = Sheets로 충분. 기존 워크플로우(엑셀→시트) 유지 |
| Claude API | 인사이트 생성 | 월 1회 데이터 갱신 시 인사이트 텍스트 생성. 정적 JSON으로 저장하여 런타임 API 호출 없음 |

Supabase 불필요 근거: 데이터 3,000행, 열람자 5명 이하, 갱신 월 1회 수동. Sheets API quota(분당 60 req/사용자)에 여유 충분.
